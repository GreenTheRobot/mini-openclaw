## 基础消融

## 数据结论

- 三个短任务在三个变体下均成功，说明当前工具链和错误恢复具有稳定性。
- `no-planning` 的平均工具调用为 10.3，低于完整系统的 11.7；平均 Token 为 46,881，低于完整系统的 54,815；平均耗时也从 18.98 秒降至 16.27 秒。
- 因此，短任务强制规划会增加开销而没有提升成功率。项目采用复杂度门控：简单任务直接执行，只有多阶段科研任务才注入 `task_list` 规划要求。
- `no-memory` 与完整系统在本组单会话任务上成功率相同，不能据此认为记忆无用；跨会话记忆的价值应由“新进程能否遵循既有项目约定”单独测试。
- 本消融只包含三个短任务，结论不能外推到论文—代码—实验—报告等长任务；后续展示应同时报告样本规模和适用范围。

## A 组：任务规划（TODO）消融

### 实验设置

本组使用一个受控的本地代码仓库，让 Agent 完成阅读说明、定位代码入口和核心模块、运行程序并撰写报告四个步骤。完整系统与 `--ablation no-planning` 各独立运行 2 次；除是否启用 TODO 规划外，模型、任务、工作区以及 `--auto-approve --no-mcp --no-multi-agent` 等运行条件完全一致。

只有当 `report.md` 写明入口、核心模块和 `accuracy=0.67`，并且 trace 同时记录到成功命令输出 `accuracy=0.67` 时，才认定任务完成。这样的判定方式可以避免 Agent 只在报告中写出正确结论、却没有实际运行程序的情况。

| 变体               |    n |  成功率 | 平均 TODO 调用 | 平均工具调用 | 平均 Token | 平均耗时(s) | trace 诊断错误 | 平均估算成本(USD) |
| ------------------ | ---: | ------: | -------------: | -----------: | ---------: | ----------: | -------------: | ----------------: |
| `none`（完整规划） |    2 | 100.00% |            9.0 |         16.0 |    128,597 |       31.54 |          0 / 2 |          0.001741 |
| `no-planning`      |    2 | 100.00% |            0.0 |          9.5 |    105,547 |       36.67 |          1 / 2 |          0.001615 |

### Trace 记录与结果

- 启用规划后，两次运行都完成了 9 次 `todo_write` 或 `update_todo` 状态更新，`diagnose` 也没有发现异常。整个任务过程因此可以按 TODO 状态回看。
- 关闭规划后，任务依然能够完成，但不会留下阶段性进度。其中一次运行出现了 `cd /workspace` 目录不存在的可恢复 bash 错误，Agent 随后自行调整并完成任务，因此该次仍计为成功。
- 关闭规划平均少调用 6.5 次工具、少消耗约 17.9% Token，却比完整规划平均多用 5.13 秒。对于这个包含多个阶段的小型任务，TODO 带来的进度约束和恢复线索，可能抵消了它本身的额外调用成本。

### 结论与边界

本组不能说明 TODO 规划提高了该确定性任务的完成率，因为两个条件都完成了全部测试；但它清楚地展示了规划模块的价值：任务过程可追踪，且完整规划的两次运行均未出现诊断错误。由于每个条件只有 2 次运行、代码仓库规模也较小，这一结果只能说明规划链路在该任务上有效，仍需在更长的科研任务中进一步验证。

## B 组：跨会话记忆消融

### 实验设置

为了验证记忆是否能在新会话中被自然利用，我们预先在独立工作区保存“用户姓名是空吧哇”的长期记忆，然后启动新的 CLI 会话，请 Agent 直接回答姓名，并明确要求不读取文件、不调用工具。完整系统会自动注入 `MEMORY.md`；`--ablation no-memory` 则跳过这一步。其余运行条件完全一致，两个条件各独立执行 2 次。

这里的“正确率”指能否在**不借助工具**的前提下正确回忆姓名。关闭记忆后的回答“未知”符合用户给出的兜底指令，因此 CLI 仍会正常结束，但在记忆能力指标上应计为未能回忆。

| 变体                       |    n | 正确回忆率 | 平均工具调用 | 平均 Token | 平均耗时(s) | trace 诊断错误 | 平均估算成本(USD) |
| -------------------------- | ---: | ---------: | -----------: | ---------: | ----------: | -------------: | ----------------: |
| `none`（自动加载长期记忆） |    2 |    100.00% |          0.0 |     21,802 |        2.65 |          0 / 2 |          0.000415 |
| `no-memory`                |    2 |      0.00% |          0.0 |     21,775 |        3.09 |          0 / 2 |          0.000145 |

### Trace 记录与结果

- 完整系统在两次新会话中都直接回答“空吧哇”，且 trace 显示工具调用数为 0。这表明答案来自会话开始时注入的长期记忆，而不是临时读取 `MEMORY.md`。
- 关闭自动记忆后，两次回答均为“未知”，同样没有调用工具。结果说明模型没有通过文件读取绕过消融条件。
- 两个条件的 Token 和耗时接近。本轮完整系统的第一次运行缓存命中较少，导致单次估算成本偏高；因此成本差异不应被解释为记忆模块本身带来的开销变化。

### 结论与边界

本组直接验证了 mini-openclaw 的长期记忆能够在新的 CLI 会话中恢复项目约定：自动加载记忆时，系统两次都正确回忆“空吧哇”；关闭自动记忆后，两次均无法给出该姓名。由于测试只使用了一条简短且无歧义的记忆、每臂仅有 2 次重复，该结论限于“记忆注入链路有效”；后续仍需用多条项目约定、冲突记忆和更长任务测试记忆检索与更新质量。

## C 组：System Prompt 消融

### 实验设置

本组要求 Agent 读取 `config.json` 中的 `seed`，修改 `hello.py`，运行脚本确认输出，并在 `final.txt` 中写入 `seed=123` 和“验证通过”。完整系统使用项目默认的 System Prompt；`--ablation minimal-prompt` 则将其替换为一句简短的命令行助手指令。

两种条件拥有完全相同的工具 schema、模型、工作区和用户任务，且都关闭 MCP 与多 Agent。也就是说，最小提示词条件仍然可以调用工具；本实验测量的是完整提示词所提供的任务指导和运行约束，而不是工具可用性。只有结果文件和 trace 中的实际命令输出都确认 `seed=123`，才认定任务完成。

| 变体                         |    n |  完成率 | 平均工具调用 | 平均 Token | 平均耗时(s) | trace 诊断错误 | 平均估算成本(USD) |
| ---------------------------- | ---: | ------: | -----------: | ---------: | ----------: | -------------: | ----------------: |
| `none`（完整 System Prompt） |    2 | 100.00% |          5.0 |     31,806 |        6.15 |          0 / 2 |          0.000243 |
| `minimal-prompt`             |    2 | 100.00% |          5.0 |     20,295 |        5.76 |          0 / 2 |          0.000211 |

### Trace 记录与结果

- 两种提示词在两次运行中都完成了读取、修改、执行和写入结果这四个步骤；每次均调用 5 次工具，`diagnose` 未发现错误。
- 最小提示词平均少消耗约 36.2% Token，平均耗时少 0.39 秒，成本也略低。这主要来自完整 System Prompt 带来的额外上下文，而不是工具调用次数的变化。
- 本组的结果表明，面对清晰、低风险的短工程任务，保留工具 schema 的简短提示词已经足以完成基本操作；完整 System Prompt 没有在完成率上表现出额外优势。

### 结论与边界

本组不能据此说明完整 System Prompt 不重要。项目默认提示词还承担安全边界、外部信息处理、复杂任务规划和失败恢复等职责，而这些能力没有在本组的无对抗短任务中受到检验。本组仅说明：在工具 schema 保留且任务足够明确时，简短提示词可以完成基础代码操作；完整提示词的价值应在安全测试、长任务和歧义任务中继续评估。

## D 组：文献链接交付硬门槛消融

### 实验设置

本组针对文献调研任务的最终交付质量进行对照。统一任务为：**“找最近一周多模态模型压缩的新论文，不用下载具体论文，给出结构化报告即可。”** 两种条件使用相同模型、相同运行日期、相同检索工具、工作目录和单 Agent 运行方式。任务只要求检索并整理近期论文信息，实验中不下载 PDF，也不进行 PDF 解析。唯一差异是最终回答出口是否启用链接交付硬门槛。

无门槛条件下，模型在完成近期论文的内容概括后可以直接结束。启用门槛的条件下，若最终答案缺少来源链接或论文卡片的必要结构，系统会拦截该次交付、基于已有检索 observation 要求重写，并在补齐后才输出最终报告。评分只读取最终回答中的 URL，不把检索工具返回的 URL 计入结果，以避免“模型看到了链接”被误判为“模型交付了链接”。

### Trace 记录与结果

- 关闭门槛后，模型能够完成最近一周论文的内容性总结，但最终答案未保留可直接点击的论文链接。该结果说明“检索结果中存在 URL”并不等价于“Agent 会在交付中主动保留 URL”:

- 启用门槛后，Trace 记录了初稿不满足交付要求后的拦截与重写过程；最终答案以结构化论文卡片呈现，并包含可点击的来源链接。因此，门槛将链接从可选的写作习惯转化为可验证的交付条件:
  ![alt text](<屏幕截图 2026-07-16 214058.png>)
  若输出中没有链接，会打回重做：
  ![alt text](image.png)

### 结论与边界

本组表明，在“找最近一周多模态模型压缩新论文并生成结构化报告”的任务中，链接硬门槛能够稳定保障最终报告包含可直接访问的来源，并维持结构化输出。

## E 组： Multi Agent Ablation

### 1. 测试口径

本次对比的是同一个图像理解任务：

> 帮我看看 `demo_project/attention_paper_parsed/images` 里的 6 张图片分别是什么内容。

对比 trace：

| 模式           | Trace                                                                         | 说明                                                                                      |
| -------------- | ----------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| 有 multi agent | `traces/ablation.jsonl` + `traces/subagents/ablation.image-analysis-1*.jsonl` | 主 Agent 启用编排，工作下沉到 Multimodal 子 Agent，并在子 Agent 内继续拆成 6 个图片子任务 |
| 无 multi agent | `traces/ablation-nomultiagent.jsonl`                                          | 单 Agent 直接 ReAct，连续调用工具完成全部图片分析和最终总结                               |

### 2. 结果概览

| 指标              |                                            有 multi agent |                            无 multi agent |
| ----------------- | --------------------------------------------------------: | ----------------------------------------: |
| 顶层运行结果      |                                             review passed |                               run success |
| 顶层总耗时        |                                                   约 739s |                                   约 441s |
| 主执行链耗时      |                                   Multimodal root 约 587s |                          单 Agent 约 441s |
| LLM 调用次数      |                                         64 次（含子任务） |                                     12 次 |
| 工具调用次数      |                                  56 次（49 成功，7 失败） |                  18 次（13 成功，5 失败） |
| 总 token          |                                                约 961,688 |                                约 215,420 |
| 最终阻塞/修订信号 | 顶层 synthesis 阶段 2 次`final_blocked`，随后 review 通过 | 1 次`final_blocked`，随后继续补证据并成功 |

结论：这次 ablation 中，multi agent 在总耗时和 token 成本上没有胜出，主要因为 `marker-004` 子任务多次调用 `paper_figure_analyze` 失败并触发了较长 fallback ，在此期间其他子任务均并行执行完毕；但它展示了更清晰的任务隔离、更强的可审查性，以及在局部子任务异常时继续综合交付的能力。

### 3. Multi Agent 技术路径

当前 multi agent 路径是轻量级编排，而不是独立系统之间的硬切分：

1. 主 Agent 先做 orchestration 判断，输出 `use_subagents`、`reason`、`main_task`、`subagents` / `assignments`。
2. 调度层按 role 选择提示词和能力边界：Research、Engineering、Multimodal。
3. 子 Agent 复用现有 `AgentLoop`、权限层、工具注册表和 trace 机制，但拥有独立任务上下文和独立 trace 文件。
4. 并行执行使用 `ThreadPoolExecutor`，每个子 Agent 完成后返回正文输出和工具证据。
5. 子 Agent 也能调用 `subagent_dispatch` 继续拆分任务，当前最多支持多层递归调度。
6. Synthesis 阶段把 `## Main Agent`、`## Multimodal Agent` 等结果块合并为最终答案。
7. Reviewer 基于子 Agent 工具调用记录和输出进行审查；如发现证据不足，会触发修订或复审。

本次实际路径如下：

```text
Main Agent
  -> Multimodal Agent: 分析 images 目录下 6 张图片
       -> subagent_dispatch: 按图片拆成 6 个 multimodal 子任务
            -> img1: marker-001.jpeg
            -> img2: marker-002.jpeg
            -> img3: marker-003.jpeg
            -> img4: marker-004.jpeg
            -> img5: marker-005.jpeg
            -> img6: marker-006.jpeg
       -> 汇总 6 张图的内容
  -> Synthesis
  -> Reviewer
```

这条路径的优点是图片之间天然独立，可以并行分析；缺点是本次主编排先派了一个宽泛 Multimodal Agent，它二次拆分 6 张图，产生了额外编排和总结成本。更理想的路径是主 Agent 直接 dispatch 6 个图片子 Agent ，这有赖于主 agent 自身的决策能力，但反映了本项目自身具有多层递归分发的能力。

### 4. 任务效果

#### 4.1 输出质量

有 multi agent 的最终审查结果为 `passed`。Reviewer 认为待审答案准确反映了 6 张图片的分析结果，和子 Agent 汇总一致，并对不确定 token 做了标注。

无 multi agent 最终也成功输出了完整报告，但中间经历了更多单链路补救：首次成稿后被判定 `insufficient_research_answer`，随后补读论文、manifest 和 grep 证据才完成最终回答。

从交付质量看，两种模式都能完成任务；multi agent 的优势在于最终结论背后有分图片 trace 和工具证据，可回溯性更强。

#### 4.2 鲁棒性

multi agent 中 6 个图片子任务里，5 个成功，`img4` 最终状态为 `partial`。尽管 `img4` 多次遇到 `paper_figure_analyze` execution error，父级 Multimodal Agent 仍然完成了综合汇总，顶层 review 通过。

无 multi agent 也遇到了一次 `paper_figure_analyze` 失败，失败对象是 `marker-006.jpeg`，随后通过再次调用恢复。单 Agent 的恢复依赖同一上下文继续变长，失败和补救都堆在一个主循环里。

因此，multi agent 对局部失败更隔离：单张图的异常不会直接污染其他图片分析；但如果某个子任务反复失败，仍会显著拖慢总耗时。

子任务明细：

| 子任务                   |    状态 |    耗时 | 主要情况                                                   |
| ------------------------ | ------: | ------: | ---------------------------------------------------------- |
| `img1 / marker-001.jpeg` | success |  约 86s | 正常完成                                                   |
| `img2 / marker-002.jpeg` | success | 约 101s | 正常完成                                                   |
| `img3 / marker-003.jpeg` | success | 约 131s | 1 次`read` 参数校验失败后自恢复                            |
| `img4 / marker-004.jpeg` | partial | 约 470s | 3 次`paper_figure_analyze` execution error，是本次实验拖尾 |
| `img5 / marker-005.jpeg` | success | 约 104s | 1 次`read` 参数校验失败后自恢复                            |
| `img6 / marker-006.jpeg` | success | 约 119s | 正常完成                                                   |

#### 4.3 性能与成本

本次 multi agent 没有体现速度优势：

- 有 multi agent 顶层耗时约 739s，Multimodal root 约 587s。
- 无 multi agent 耗时约 441s。
- multi agent 总 token 约 96.2 万，是无 multi agent 约 21.5 万的 4.5 倍。

主要原因不是并行机制无效，而是这次实际路径存在两个成本放大点：

1. 两级拆分：Main -> Multimodal root -> 6 个图片子 Agent，多了一层中间 agent 的规划、读取和汇总。
2. 慢子任务拖尾：`img4` 耗时约 470s，消耗约 42 万 token，是总成本的主要来源。

并行系统的总 wall time 由最慢子任务决定；当某个子任务反复失败时，multi agent 会保留可隔离性，但速度优势会被拖尾吃掉。

### 5. 对比结论

这次 ablation 支持以下判断：

| 维度     | 判断                                                                                         |
| -------- | -------------------------------------------------------------------------------------------- |
| 能力覆盖 | multi agent 可以正确识别图像类任务并启用 Multimodal 路径                                     |
| 任务拆解 | 子 Agent 能进一步把 6 张图片拆成独立并行任务，技术路径成立                                   |
| 质量     | multi agent 最终 review passed，输出可追溯性更好                                             |
| 鲁棒性   | 局部失败被限制在單个圖片子任务内，父级不受污染，仍能综合交付                                 |
| 速度     | 由最慢子 agent 限制，显著小于所有任务总和；但单个子 agent 任务失败时外在表现可能慢于单 Agent |
| 成本     | token 成本明显更高                                                                           |

整体结论：multi agent 的价值主要体现在复杂任务的结构化拆解、证据隔离和审查闭环，以及即时性能提升。对于 6 张图片这种天然可并行任务，技术方向是正确的；但当前实现需要减少中间层、限制慢任务重试、复用工具结果，才能把并行优势真正转化为 wall time 和成本优势。
