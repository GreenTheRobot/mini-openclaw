###  数据结论

- 三个短任务在三个变体下均成功，说明当前工具链和错误恢复具有稳定性。
- `no-planning` 的平均工具调用为 10.3，低于完整系统的 11.7；平均 Token 为 46,881，低于完整系统的 54,815；平均耗时也从 18.98 秒降至 16.27 秒。
- 因此，短任务强制规划会增加开销而没有提升成功率。项目采用复杂度门控：简单任务直接执行，只有多阶段科研任务才注入 `task_list` 规划要求。
- `no-memory` 与完整系统在本组单会话任务上成功率相同，不能据此认为记忆无用；跨会话记忆的价值应由“新进程能否遵循既有项目约定”单独测试。
- 本消融只包含三个短任务，结论不能外推到论文—代码—实验—报告等长任务；后续展示应同时报告样本规模和适用范围。

## A 组：任务规划（TODO）消融

### 实验设置

本组使用一个受控的本地代码仓库，让 Agent 完成阅读说明、定位代码入口和核心模块、运行程序并撰写报告四个步骤。完整系统与 `--ablation no-planning` 各独立运行 2 次；除是否启用 TODO 规划外，模型、任务、工作区以及 `--auto-approve --no-mcp --no-multi-agent` 等运行条件完全一致。

只有当 `report.md` 写明入口、核心模块和 `accuracy=0.67`，并且 trace 同时记录到成功命令输出 `accuracy=0.67` 时，才认定任务完成。这样的判定方式可以避免 Agent 只在报告中写出正确结论、却没有实际运行程序的情况。

| 变体 | n | 成功率 | 平均 TODO 调用 | 平均工具调用 | 平均 Token | 平均耗时(s) | trace 诊断错误 | 平均估算成本(USD) |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `none`（完整规划） | 2 | 100.00% | 9.0 | 16.0 | 128,597 | 31.54 | 0 / 2 | 0.001741 |
| `no-planning` | 2 | 100.00% | 0.0 | 9.5 | 105,547 | 36.67 | 1 / 2 | 0.001615 |

### Trace 记录与结果

- 启用规划后，两次运行都完成了 9 次 `todo_write` 或 `update_todo` 状态更新，`diagnose` 也没有发现异常。整个任务过程因此可以按 TODO 状态回看。
- 关闭规划后，任务依然能够完成，但不会留下阶段性进度。其中一次运行出现了 `cd /workspace` 目录不存在的可恢复 bash 错误，Agent 随后自行调整并完成任务，因此该次仍计为成功。
- 关闭规划平均少调用 6.5 次工具、少消耗约 17.9% Token，却比完整规划平均多用 5.13 秒。对于这个包含多个阶段的小型任务，TODO 带来的进度约束和恢复线索，可能抵消了它本身的额外调用成本。


### 结论与边界

本组不能说明 TODO 规划提高了该确定性任务的完成率，因为两个条件都完成了全部测试；但它清楚地展示了规划模块的价值：任务过程可追踪，且完整规划的两次运行均未出现诊断错误。由于每个条件只有 2 次运行、代码仓库规模也较小，这一结果只能说明规划链路在该任务上有效，仍需在更长的科研任务中进一步验证。

## B 组：跨会话记忆消融

### 实验设置

为了验证记忆是否能在新会话中被自然利用，我们预先在独立工作区保存“用户姓名是空吧哇”的长期记忆，然后启动新的 CLI 会话，请 Agent 直接回答姓名，并明确要求不读取文件、不调用工具。完整系统会自动注入 `MEMORY.md`；`--ablation no-memory` 则跳过这一步。其余运行条件完全一致，两个条件各独立执行 2 次。

这里的“正确率”指能否在**不借助工具**的前提下正确回忆姓名。关闭记忆后的回答“未知”符合用户给出的兜底指令，因此 CLI 仍会正常结束，但在记忆能力指标上应计为未能回忆。

| 变体 | n | 正确回忆率 | 平均工具调用 | 平均 Token | 平均耗时(s) | trace 诊断错误 | 平均估算成本(USD) |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `none`（自动加载长期记忆） | 2 | 100.00% | 0.0 | 21,802 | 2.65 | 0 / 2 | 0.000415 |
| `no-memory` | 2 | 0.00% | 0.0 | 21,775 | 3.09 | 0 / 2 | 0.000145 |

### Trace 记录与结果

- 完整系统在两次新会话中都直接回答“空吧哇”，且 trace 显示工具调用数为 0。这表明答案来自会话开始时注入的长期记忆，而不是临时读取 `MEMORY.md`。
- 关闭自动记忆后，两次回答均为“未知”，同样没有调用工具。结果说明模型没有通过文件读取绕过消融条件。
- 两个条件的 Token 和耗时接近。本轮完整系统的第一次运行缓存命中较少，导致单次估算成本偏高；因此成本差异不应被解释为记忆模块本身带来的开销变化。

### 结论与边界

本组直接验证了 mini-openclaw 的长期记忆能够在新的 CLI 会话中恢复项目约定：自动加载记忆时，系统两次都正确回忆“空吧哇”；关闭自动记忆后，两次均无法给出该姓名。由于测试只使用了一条简短且无歧义的记忆、每臂仅有 2 次重复，该结论限于“记忆注入链路有效”；后续仍需用多条项目约定、冲突记忆和更长任务测试记忆检索与更新质量。

## C 组：System Prompt 消融

### 实验设置

本组要求 Agent 读取 `config.json` 中的 `seed`，修改 `hello.py`，运行脚本确认输出，并在 `final.txt` 中写入 `seed=123` 和“验证通过”。完整系统使用项目默认的 System Prompt；`--ablation minimal-prompt` 则将其替换为一句简短的命令行助手指令。

两种条件拥有完全相同的工具 schema、模型、工作区和用户任务，且都关闭 MCP 与多 Agent。也就是说，最小提示词条件仍然可以调用工具；本实验测量的是完整提示词所提供的任务指导和运行约束，而不是工具可用性。只有结果文件和 trace 中的实际命令输出都确认 `seed=123`，才认定任务完成。

| 变体 | n | 完成率 | 平均工具调用 | 平均 Token | 平均耗时(s) | trace 诊断错误 | 平均估算成本(USD) |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `none`（完整 System Prompt） | 2 | 100.00% | 5.0 | 31,806 | 6.15 | 0 / 2 | 0.000243 |
| `minimal-prompt` | 2 | 100.00% | 5.0 | 20,295 | 5.76 | 0 / 2 | 0.000211 |

### Trace 记录与结果

- 两种提示词在两次运行中都完成了读取、修改、执行和写入结果这四个步骤；每次均调用 5 次工具，`diagnose` 未发现错误。
- 最小提示词平均少消耗约 36.2% Token，平均耗时少 0.39 秒，成本也略低。这主要来自完整 System Prompt 带来的额外上下文，而不是工具调用次数的变化。
- 本组的结果表明，面对清晰、低风险的短工程任务，保留工具 schema 的简短提示词已经足以完成基本操作；完整 System Prompt 没有在完成率上表现出额外优势。

### 结论与边界

本组不能据此说明完整 System Prompt 不重要。项目默认提示词还承担安全边界、外部信息处理、复杂任务规划和失败恢复等职责，而这些能力没有在本组的无对抗短任务中受到检验。本组仅说明：在工具 schema 保留且任务足够明确时，简短提示词可以完成基础代码操作；完整提示词的价值应在安全测试、长任务和歧义任务中继续评估。

## D 组：多 Agent 协作消融（真实 CLI，2026-07-16）

### 实验设置

本地 fixture 包含四份材料：token pruning 与 35% FLOPs 的方法声称、8-bit quantization 与 calibration drift 风险、OOD 泛化尚未验证的限制，以及一段诱导系统跳过限制并宣称泛化的未可信指令；代码侧包含 `ImageEncoder`、`quantize`、`evaluate` 三个模块及评估、基准、回归三个脚本。

默认条件允许主 Agent 调度 Research Agent、Engineering Agent 和 Reviewer：Research Agent 负责整理文档证据与限制，Engineering Agent 负责检查代码并实际执行三个脚本，主 Agent 汇总后交由 Reviewer 进行结论审查。对照条件使用 `--no-multi-agent`，由单 Agent 执行完全相同的用户任务。两种条件使用相同的模型、fixture、用户提示和 `--auto-approve --no-mcp` 参数，各独立运行 2 次；唯一差异是是否启用多 Agent。

完成标准同时要求最终答复覆盖研究、代码、运行与安全限制事实，并要求合并后的 trace 实际出现 `score=0.92`、`latency_ms=12.5`、`regression=pass` 三个成功命令输出。评分器只处理 `ImageEncoder` / `image encoder`、以及“未验证” / “尚未验证”等等价表述，不以答复长度作为完成依据。

| 变体 | n | 证据覆盖与命令核验 | 子 Agent trace | 平均工具调用 | 平均 Token | 平均耗时(s) | trace 诊断错误 | 平均估算成本(USD) |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 默认多 Agent | 2 | 100.00% | 2 / 2 次运行（共 4 条） | 35.5 | 154,142 | 91.92 | 1 次 | 0.005146 |
| `--no-multi-agent` | 2 | 100.00% | 0 | 28.5 | 193,959 | 75.66 | 2 次 | 0.004721 |

### Trace 记录与结果

- 两种条件均在两次独立运行中完成了研究证据整理、代码路径核验、三个脚本执行和谨慎结论输出；多 Agent 条件的每次运行均生成 Research 与 Engineering 子 trace。
- 在相同完成率和证据覆盖下，多 Agent 的平均 Token 为 154,142，比单 Agent 的 193,959 少约 **20.5%**。这表明角色拆分减少了单一上下文反复混合文档、代码和审查信息的开销。
- 多 Agent 的代价同样清晰：平均工具调用更多（35.5 对 28.5），平均耗时更长（91.92 秒对 75.66 秒，约高 21.5%），估算成本也略高。它并非在所有指标上都优于单 Agent。
- 合并父 trace 与子 trace 后，多 Agent 共记录 1 次诊断错误，低于单 Agent 的 2 次；两边均未影响最终交付。父 trace 只记录协调事件，因此成本、工具调用与错误统计均以合并 trace 为准。

### 结论与边界

本组给出一个明确但有限的正向结果：**在该高信息密度、跨研究—工程—审查任务中，多 Agent 能够在保持 100% 完成率和相同证据核验标准的前提下节省一定的Token消耗。**这正体现了 Research、Engineering 与 Reviewer 分工在长链信息整合任务中的价值。
